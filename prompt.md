# Pick Background Improvement Loop

> 15분 간격 자동 실행. 매 사이클 시작 시 반드시 CLAUDE.md 참조.
> 브랜치: `background-improve`에서만 작업. main은 절대 건드리지 않음.

## 서비스 핵심 (절대 잊지 말 것)

**Pick = 강사와 학생의 실시간 상호작용 플랫폼**

- **핵심 가치**: 수업 중 강사↔학생 소통. 이것이 모든 기능의 존재 이유.
- **강사**: 학생 참여를 이끌고, 학습 결과 데이터를 확인하고 활용할 수 있어야 함
- **학생**: 쉽고 빠르게 수업에 참여하고, 즉각적 피드백을 받아야 함
- **절대 안 됨**: 상호작용과 무관한 기능 추가, 핵심에서 벗어난 방향 전환
- **admin 데이터 활용**: 수업 기록, 참여율 분석, 학생별 참여도, 질문 결과 통계 → 강사가 수업 개선에 활용 가능해야 함

## 브랜치 규칙

- **작업 브랜치**: `background-improve` (이 브랜치에서만 커밋/푸시)
- **서브에이전트 worktree**: 사용 가능하나, 작업 완료 후 반드시 `background-improve`에 merge
- **main 브랜치**: 절대 직접 커밋/푸시하지 않음 (사용자가 직접 머지)
- **다른 브랜치 생성 금지**: worktree 임시 브랜치 외에 새 브랜치 만들지 않음
- **하나의 브랜치에 계속 커밋**: 매 사이클마다 커밋하고 push. 머지는 사용자가 리뷰 후 직접
- **충돌 발생 시**: 반드시 해결 후 merge. 해결 못하면 해당 작업 버리고 다음으로

### 토스(Toss) 모션 철학 (참고)
> 디자인 철학/원칙 참고는 법적으로 문제없음 (Apple HIG, Material Design 참고와 동일).
> 코드/브랜드 에셋 복사는 금지. 원칙만 우리 방식으로 재해석하여 적용.

1. **목적 있는 모션만** — 장식 모션 0, 피드백/가이드/상태전환에만
2. **터치 피드백은 신성** — 버튼 press, 로딩, 수량 조절 모두 모션
3. **공유 easing 라이브러리** — enter: `[0.33, 1, 0.68, 1]`, exit: `[0.32, 0, 0.67, 0]`, move: `[0.65, 0, 0.35, 1]`
4. **입퇴장 안무** — 요소가 그냥 나타나고 사라지지 않음. 의도를 가지고 등장/퇴장
5. **진화 경로**: Framer Motion CSS → 선별적 Lottie(축하/빈상태) → Rive(인터랙티브, 미래)

### 모션 & 캐릭터 (핵심 UX 요소)

> 레퍼런스: 토스(부드러운 전환), 듀오링고(캐릭터 생동감), Linear(절제된 모션)

**목표**: 화려하지 않지만 "살아있는" 느낌. 앱이 숨 쉬는 것처럼 자연스럽게.

- **Framer Motion**: 모든 상태 전환에 부드러운 모션 필수
  - 페이지 전환: fade + subtle slide
  - 카드 등장: stagger (0.03~0.05s 간격)
  - 데이터 변경: spring 애니메이션으로 숫자 카운트업
  - 리스트 추가/삭제: AnimatePresence로 자연스럽게
- **Lottie**: 상업적 무료 범위 내 자유 활용 (LottieFiles CDN)
- **Pick 마스코트 활용 (적극적으로)**:
  - 대기 화면: 마스코트가 산책하거나 장난치는 애니메이션
  - 빈 상태: 마스코트가 안내하거나 기다리는 포즈
  - 퀴즈 정답: 마스코트 축하 (현재 컨페티 + 추가 가능)
  - 연승 중: 마스코트가 응원하는 모습
  - 이스터에그: 특정 조건에서 마스코트가 깜짝 등장
  - 에러 상태: 마스코트가 당황하거나 미안해하는 포즈
- **마이크로 인터랙션 (반드시)**:
  - 버튼 탭: scale bounce + haptic (가능 시)
  - 투표 선택: 선택지가 살짝 떨리며 확인 느낌
  - 손들기: 손 아이콘이 흔들리는 모션
  - 채팅 전송: 메시지가 날아가는 느낌
  - 참여자 입장: Avatar가 통통 튀며 등장
- **절제 기준**:
  - 수업 진행을 방해하면 안 됨
  - 모든 모션은 400ms 이하
  - `prefers-reduced-motion` 존중
  - 화려함 < 자연스러움 < 의미 있음

## 기획 & 신규 기능

매 사이클마다 기획적 관점에서도 고민:
- **강사에게 도움이 되는 기능**: 수업 진행 편의, 학생 관리, 결과 분석
- **학생에게 도움이 되는 기능**: 참여 경험, 피드백, 학습 동기부여
- 가치 있다고 판단되면 **한 사이클에 하나씩** 도전 (여러 개도 OK, 단 각각 완결)

- 학생 화면은 모바일 퍼스트 — 한 손 조작, 1초 안에 뭘 해야 하는지 파악

### 기능 아이디어 검토 기준
- "이 기능이 없으면 수업이 불편한가?" → Yes면 필수
- "이 기능이 있으면 수업이 더 재미있어지는가?" → Yes면 우선
- "이건 쿨해 보이지만 실제로 쓸까?" → No면 보류

## 백엔드 & 데이터 규칙

### 아키텍처: React + Firebase (서버리스)
- 별도 백엔드 서버 없음. React에서 Firebase SDK로 직접 읽기/쓰기
- **Mock 데이터 절대 금지**: 하드코딩된 가짜 데이터로 UI 채우지 않기
- **모든 데이터는 Firebase Realtime DB**에서 읽고 쓰기 (`onValue`, `set`, `push`, `update`)
- **테이블(경로) 추가**: 새 기능에 필요하면 자유롭게 추가
  - `database.rules.json`에 규칙 반드시 추가
  - 기존 데이터 구조와 일관성 유지 (CLAUDE.md 아키텍처 참고)
- **데모 데이터 갱신**: 구조 변경 시 `scripts/seed-demo.mjs` 업데이트 후 재실행
  - `node scripts/seed-demo.mjs`로 기존 데이터 삭제 + 새 데이터 시드
- **기존 테이블 구조 변경 시**: database.rules.json도 함께 수정

### 프론트-백 연동 체크
- 새 훅 만들 때: `onValue`/`onChildAdded` 실시간 구독 사용
- `useEffect` cleanup에서 Firebase 리스너 해제 필수
- 데이터 쓰기: `set`/`update`/`push` 직접 호출
- 에러 처리: try-catch로 감싸고, 사용자에게 실패 피드백
- **매 사이클 검증**: Playwright로 실제 데이터가 Firebase에 저장/조회되는지 확인

## 회귀 방지 규칙

- **파일 수정 전 반드시 Read** — 현재 상태를 모르고 수정하면 다른 사람의 작업을 덮어쓸 수 있음
- **큰 변경은 서브에이전트** — 5개 파일 이상 수정 시 서브에이전트 사용 (컨텍스트 보호)
- **하나의 관심사만** — 한 커밋에 여러 종류의 변경 섞지 않기 (UI + 기능 + 버그 → 별도 커밋)
- **spring 값은 프리셋만** — design-tokens.js에 정의된 5개 프리셋(default/gentle/bouncy/stiff/smooth) 외 사용 금지
- **색상은 slate 계열만** — bg-indigo-600 CTA 금지, bg-amber/emerald 장식 금지
- **text-slate-300은 장식에만** — 읽어야 하는 텍스트는 최소 text-slate-500 (dark: text-slate-400)
- **새 컴포넌트는 memo 고려** — props 받는 리스트 아이템, 차트, 투표 컴포넌트 → React.memo
- **새 파일은 200줄 이하** — 초과 시 서브컴포넌트로 분리

### 스태프 모드 (Staff Mode) — 우선 구현
현장 강의에는 강사 1명 + 운영 스태프 5~10명이 함께 참여. 스태프도 앱을 써야 함.

#### 핵심 컨셉
스태프는 "강사의 눈과 손" — 학생 질문에 답변하고, 손들기 확인 후 자리에 가서 도움주고, 채팅으로 소통. 강사가 수업 진행에 집중할 수 있도록 지원하는 역할.

#### 스태프 계정
- 강사(admin)가 더보기에서 스태프 초대 (별도 역할: `staff`)
- 데모 로그인: "스태프 데모로 둘러보기" (이미 구현)
- Firebase rules: role에 'staff' 허용 (이미 구현)

#### 스태프 전용 화면 (강사 3패널과 다르게!)
스태프 화면은 **모바일 최적화** — 스태프는 강의장을 돌아다니며 학생 도움. 태블릿/폰으로 사용.

**스태프 화면 구성 (단일 컬럼, 모바일 퍼스트):**
```
[헤더] 강의명 + "스태프" 배지 + 접속자 수
[탭 바] 질문 | 손들기 | 채팅 | 참여자
[콘텐츠 영역]
  - 질문 탭: 긴급 질문 + 수업 질문 통합 리스트. 확인/답변 완료 가능
  - 손들기 탭: 손든 학생 목록 + "도움 완료" 버튼 (해제)
  - 채팅 탭: 채팅 (senderType: 'staff', "스태프" 배지)
  - 참여자 탭: 접속 학생 목록 (참여도 표시)
```

#### 스태프 권한 (명확한 분리)
| 기능 | 강사 | 스태프 |
|------|------|--------|
| 질문 활성화/중지 | ✅ | ❌ |
| 질문 추가/삭제/복제 | ✅ | ❌ |
| 세션 시작/종료 | ✅ | ❌ |
| 발표 모드 | ✅ | ❌ |
| 타이머 설정 | ✅ | ❌ |
| 게임 모드(룰렛/추첨) | ✅ | ❌ |
| 팀 대항전 관리 | ✅ | ❌ |
| 긴급 질문 확인/삭제 | ✅ | ✅ |
| 수업 질문 답변 완료 | ✅ | ✅ |
| 손들기 해제 (도움 완료) | ✅ | ✅ |
| 채팅 참여 | ✅ | ✅ (스태프 배지) |
| 참여자 목록 보기 | ✅ | ✅ |
| 실시간 투표 결과 보기 | ✅ | ✅ (읽기 전용) |

#### 스태프 식별
- 채팅: "스태프" 배지 (이미 구현)
- 학생 화면: 스태프의 답변/채팅에 "강사 팀" 느낌
- 어드민 화면: 스태프 목록 관리 (더보기 탭)

#### 구현 순서
1. ✅ Firebase rules staff 역할 추가
2. ✅ 데모 로그인 추가
3. ✅ 채팅 스태프 배지
4. ✅ 프로필 역할 표시
5. ⬜ AdminPage에서 role === 'staff' 분기 → StaffPage 컴포넌트
6. ⬜ StaffPage: 단일 컬럼 + 4탭 (질문/손들기/채팅/참여자)
7. ⬜ 스태프 질문 통합 뷰 (긴급+수업 질문 합쳐서 시간순)
8. ⬜ 스태프 손들기 뷰 ("도움 완료" 버튼)
9. ⬜ 강사 더보기에서 스태프 관리 UI

### 장기 투표 모드 (Persistent Poll)
강의 중 30분~1시간 동안 투표를 열어두고, 학생이 언제든 참여 가능.

- **학생 화면**: 상단에 플로팅 배너로 "투표 진행 중" 표시. 탭하면 투표 화면 열림
- **강사 화면**: 실시간으로 투표 결과 변동 확인 가능
- **용도**: 실습 주제 결정, 발표 순서 투표, 의견 수렴 등
- **타이머**: 강사가 설정한 시간 후 자동 종료 (또는 수동 종료)
- **기존 투표와 차이**: 기존은 질문 활성화→즉시 응답. 이건 백그라운드로 열어두고 수업 진행

### 새 모드 아이디어 (구현 가능한 것부터)
매 사이클에서 하나씩 기획→구현 시도. 서브에이전트로 레퍼런스 조사 병행.

- **빠른 응답 모드**: 가장 먼저 정답 맞춘 학생이 승리 (속도 배틀)
- **팀 대항전**: 학생들을 자동 팀 분배 → 팀별 점수 경쟁
- **찬반 토론**: 주제 제시 → 찬성/반대 선택 + 한 줄 의견 → 실시간 토론 시각화
- **순위 맞추기**: 항목들의 순서를 맞추는 문제 (드래그로 정렬)
- **빈칸 채우기**: 문장에 빈칸 → 학생이 채우기 (주관식 변형)
- **포인트 베팅**: 자기 점수를 걸고 답변 (확신도 표현)
- **스피드 퀴즈**: 연속 문제 폭탄 (10초 제한, 연속 정답 콤보 보너스)
- **이미지 투표**: 텍스트 대신 이미지로 선택지 (시각적 설문)
- **감정 온도계**: 슬라이더로 0~100 감정/의견 강도 표현 (평균값 시각화)

### 기존 기능에 재미 추가
- **연승 보너스**: 3연속 정답 시 특별 이펙트 + 추가 점수
- **최다 참여상**: 세션 종료 시 가장 많이 참여한 학생 하이라이트

## Playwright 기능 검증 (매 사이클 필수)

스크린샷만 찍지 말고, **실제로 클릭하고 동작을 확인**:

### 강사 화면 검증
- 데모 로그인 → 세션 입장 → 질문 활성화 → 투표 결과 확인
- 질문 추가 폼 열기 → 입력 → 제출 → 목록에 나오는지 확인
- 타이머 팝업 열기/닫기, 채팅 팝업 열기/닫기
- 패널 접기/펼치기, 아코디언 동작
- 세션 종료 → 완료 상태에서 결과 보기

### 학생 화면 검증
- 세션 링크 접속 → 닉네임 입력 → 참여
- 투표 선택 → 확인 → 결과 대기
- 하단 바 (손들기, 긴급질문) 동작
- 리액션 전송
- 모바일 뷰포트(390x844)에서 레이아웃 확인

### 검증 규칙
- **클릭 → 결과 확인**: 버튼 눌렀는데 반응 없으면 버그 → 즉시 수정
- **빈 상태 확인**: 데이터 없을 때 화면이 어떻게 보이는지
- **에러 상태 확인**: 잘못된 입력, 네트워크 끊김 등
- 신규 기능 추가 시 → 반드시 Playwright로 전체 플로우 테스트

## 성능 & 코드 품질 (매 사이클 체크)

### 성능 최적화
- **React.memo**: 자주 리렌더되는 컴포넌트 감싸기
- **useCallback/useMemo**: props로 넘기는 함수/계산값 메모이제이션
- **코드 스플리팅**: 새 페이지/무거운 컴포넌트는 React.lazy
- **Firebase 구독 정리**: useEffect cleanup에서 리스너 해제
- **이미지/에셋**: 불필요한 에셋 제거, SVG 최적화

### 데드 코드 제거
- 사용되지 않는 import, 변수, 함수, 컴포넌트 → 즉시 삭제
- 주석 처리된 코드 → 삭제 (git에 히스토리 있으므로)
- 사용되지 않는 파일 → 삭제 (예: PinggoMascot.jsx 같은 구 컴포넌트)
- `grep -r "import.*from" src/ | sort` 로 미사용 import 스캔

### 데이터 규칙 (강화)
- **하드코딩 데이터 절대 금지**: UI에 표시되는 모든 데이터는 Firebase에서 실시간으로 가져와야 함
- 데모/테스트용 하드코딩도 금지 — seed-demo.mjs에서 Firebase에 넣고 가져다 쓰기
- 통계, 카운트, 참여자 수 등 모든 숫자는 실시간 계산

### 디자인 시스템 준수 (지속적)
- 매 사이클마다 DESIGN_SYSTEM.md를 참조하여 UI/UX 개선
- 새 컴포넌트 만들 때 → 디자인 시스템 프리셋 값만 사용 (spring, color, spacing, typography)
- 기존 화면에서 디자인 시스템 벗어난 값 발견 → 수정
- 목표: Apple/Toss 수준의 아름답고 절제된 UI
