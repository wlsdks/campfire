## 서비스 정체성

**Pick = 현장 강의 전용 실시간 학습 플랫폼**

### 핵심 가치 (5대 축)
1. **소통** — 강사↔학생, 스태프↔학생, 학생↔학생(수업 질문) 실시간 양방향
2. **학습 기능** — 다양한 질문 유형(객관식/OX/퀴즈/토론/순위/빈칸/스케일), 퀴즈 점수, 자신감 측정
3. **부드러운 학습 연계** — 질문→투표→결과→다음 질문이 끊김 없이 자연스럽게 흐름
4. **재미** — 게임(룰렛/추첨/리더보드), 업적, 연승, 팀 대항전, 리액션
5. **인사이트** — 참여율 추이, 어려운 질문 분석, 학생별 성과, CSV 내보내기

### 사용자
- **강사** — 수업 진행 중 원클릭 조작. 프로젝터 + 노트북
- **스태프** — 강의장 돌아다니며 학생 도움. 태블릿/폰
- **학생** — 모바일 한 손 조작. 1초 안에 뭘 해야 하는지 파악
- **전자칠판** — `/live?s=세션코드` 읽기전용 실시간 결과

## 디자인 최우선 원칙

> Apple/Toss/Linear 수준의 UI/UX. "AI가 만든 것처럼 보이면 실패."

### Apple/Toss 디자인 철학

> **상세 원칙·규칙·Anti-AI 체크리스트·컴포넌트 레시피는 모두 `DESIGN_SYSTEM.md`에 정의되어 있다.**
> 매 사이클 시작 시 반드시 `Read DESIGN_SYSTEM.md` 하여 최신 규칙을 숙지한 뒤 작업할 것.

핵심 요약 (상세는 DESIGN_SYSTEM.md):
1. **빼는 디자인** — 요소를 추가하지 말고 불필요한 것을 제거
2. **여백이 디자인** — 정보 밀도 낮추고 호흡. 의심되면 늘려라
3. **타이포가 위계** — 색상 대신 크기/두께/투명도로 구분
4. **모션은 의미만** — 피드백/전환/관계. 장식 모션 0
5. **일관성 = 신뢰** — 같은 패턴은 항상 같게
6. **한국어 UX** — 토스/카카오뱅크 스타일. 아이콘+텍스트 라벨 필수

### 매 사이클 디자인 자문 (체크리스트 전에 먼저 하기)
```
1. 수정한 화면을 Playwright 스크린샷으로 확인한다
2. "빼도 되는 요소가 있는가?" → 빼라
3. "여백이 충분한가?" → 의심되면 늘려라
4. "숫자/라벨 크기 위계가 토스처럼 명확한가?" → 숫자 크게, 라벨 작게
5. "색상이 2색(slate + indigo) 안에 드는가?" → 장식 색상 제거
```

### 매 사이클 기계적 체크리스트
```
□ 한 화면에 3색 이상? → 2색으로 줄이기
□ CTA가 bg-indigo-600? → bg-slate-900 사용
□ 아이콘이 색상 원형 배경 안? → bare 아이콘
□ 숫자가 라벨보다 작은가? → 숫자를 3x+ 크게
□ 여백이 부족한가? → 섹션 간 space-y-5 이상
□ 보더+그림자 동시 사용? → 택 1
□ transition에 duration 없는가? → transition-colors duration-150
□ heading에 tracking-tight 없는가? → 추가
□ spring이 프리셋(300/25, 200/20, 400/22, 500/30) 아닌가? → 교체
□ 빈 상태에 마스코트 없는가? → PickMascot 추가
```

### 확정된 규칙 (절대 위반 금지)
- `transition-colors duration-150` — 모든 버튼/인터랙티브 (입력 필드는 transition-all OK)
- `tracking-tight` — text-xl+ font-bold 모든 헤딩
- spring 프리셋 4종만 사용 (default/gentle/bouncy/stiff)
- 장식 색상 0 — slate 모노크롬 + indigo 악센트(차트/진행바) + 기능색(red/emerald)만
- 빈 상태 = PickMascot + 안내 텍스트 (아이콘만 사용 금지)
- shadow-sm 카드에서 border 제거 (이중 깊이 단서 금지)
- 채팅 메시지 slide-in 애니메이션 필수
- 숫자 변경 시 motion.span key={value} bounce

## 모드 전환 UX 강화 (v3 중점)

> 강사가 모드를 전환할 때 학생/프레젠터 화면이 매끄럽게 바뀌어야 한다.
> 현재는 그냥 컴포넌트가 뚝 바뀌는데, 이걸 **연출**로 만들어야 한다.

### 전환 흐름 & 연출
```
대기(waiting) ←→ 질문(poll) ←→ 결과(result) ←→ 게임(roulette/slot/pinball)
     ↕                                              ↕
  리더보드(leaderboard)                        쉬는시간(break)
```

### 전환 애니메이션 가이드
| 전환 | 현재 | 목표 |
|------|------|------|
| 대기 → 질문 | 즉시 교체 | slide-up + fade. 질문 카드가 아래서 올라옴 |
| 질문 → 결과 | 즉시 교체 | 투표 UI fold → 결과 바 grow-in |
| 결과 → 게임 | 즉시 교체 | scale-out → 게임 scale-in (드라마틱) |
| 게임 → 대기 | 즉시 교체 | fade-out → 대기 fade-in |
| 어디서든 → 리더보드 | 즉시 교체 | curtain-reveal 또는 slide-down |

### 모드 전환 원칙
1. **AnimatePresence mode="wait"** — 이전 모드 exit 완료 후 다음 enter
2. **exit 애니메이션은 짧게** (0.15-0.2s), enter는 약간 길게 (0.25-0.35s)
3. **방향성** — 앞으로 가면 slide-left/up, 뒤로 가면 slide-right/down
4. **게임 진입은 특별하게** — scale + opacity로 "무대 등장" 느낌
5. **학생 화면도 동시 전환** — Firebase 상태 변경 감지 → 같은 애니메이션

### 래퍼런스
- Keynote/PowerPoint 슬라이드 전환 (매직 무브)
- Kahoot 질문 전환 (카운트다운 → 질문 reveal)
- Apple TV+ 콘텐츠 카드 전환

## 모바일 디자인 개선 가이드

> PWA이지만 네이티브 앱 수준의 UX를 목표로 한다.
> 래퍼런스: 토스, 당근마켓, 카카오톡, Apple HIG, Linear
> 디자인 시스템 상세는 `DESIGN_SYSTEM.md §21` 참조.

### 핵심 철학 (매 사이클 반드시 숙지)
1. **부드러움과 역동적 애니메이션** — spring 물리, 자연스러운 전환, 의미있는 모션
2. **사용성과 편리함** — 1초 안에 뭘 해야 하는지 파악, 복잡하지 않음
3. **적절한 공백의 활용** — 밀도 < 여유, 공백이 디자인이다
4. **글자 크기 위계** — 모바일은 데스크탑보다 +1~2px, 큰 숫자+작은 라벨
5. **배경 대비로 섹션 구분** — border/shadow 대신 bg 색상 차이 (토스/당근 패턴)
6. **전체 화면 모달** — 모바일에서 모든 패널은 inset-0 (앱 느낌)

### 매 사이클 모바일 체크리스트
```
1. Playwright 390x844 뷰포트에서 스크린샷 찍기
2. "이 화면이 앱처럼 보이는가?" — 웹 느낌 요소 제거
3. "글씨가 충분히 큰가?" — 본문 15-16px, 제목 17-20px
4. "터치 타겟이 44px+ 인가?" — 모든 버튼/행
5. "불필요한 border가 있는가?" — 배경 대비로 대체
6. "모달이 전체 화면인가?" — inset-0 on mobile
7. "여백이 넉넉한가?" — px-5, py-4 이상
8. "애니메이션이 자연스러운가?" — spring 물리, 과하지 않게
```

### 웹용 vs 모바일용 디자인 토큰
> 상세 정의는 DESIGN_SYSTEM.md에. 여기는 핵심 차이만.

| 토큰 | 웹 (1024px+) | 모바일 (<768px) |
|------|-------------|----------------|
| 본문 크기 | text-sm (14px) | text-[15px]~text-base |
| 제목 크기 | text-xl~2xl | text-base~text-lg |
| 섹션 제목 | text-sm semibold | text-[16-17px] semibold |
| 카드 패딩 | p-5 | p-5 (동일) |
| 섹션 간격 | space-y-4~5 | space-y-2~3 (bg 대비) |
| 모달 | centered 420x600 | 전체 화면 inset-0 |
| 섹션 구분 | border + shadow | 배경색 대비 (no border) |
| 헤더 | border-b | backdrop-blur만 |
| 탭 바 | 없음 (3패널) | 하단 56px + safe area |
| 아이콘 크기 | 14-18px | 18-24px |

## 기능 연계 점검 (매 사이클 필수)

> 기능이 개별적으로 잘 동작해도, 서로 연결이 안 되면 UX가 끊긴다.
> 매 사이클마다 아래 흐름을 Playwright로 실제 테스트하고, 끊기는 곳이 있으면 즉시 수정.

### 학생 전체 흐름
```
JoinPage → WaitingPage → VotePage(투표) → VoteConfirm → 결과 대기 → 다음 질문
                                                            ↕
                                          StudentLiveResults (실시간 결과)
                                          LeaderboardPage (퀴즈 시)
                                          SessionSummaryCard (수업 종료 시)
```
- 질문 전환이 부드러운가? (AnimatePresence mode="wait")
- 투표 후 실시간 결과가 바로 보이는가?
- 퀴즈 정답 공개 시 정답/오답 피드백이 즉각적인가?
- 수업 종료 시 요약 카드가 자연스럽게 나타나는가?

### 강사↔학생 상호작용
```
강사: 질문 활성화 → 학생: 자동 투표 화면 전환
강사: 정답 공개   → 학생: 정답/오답 + 점수 표시
강사: 다음 질문   → 학생: 부드러운 질문 전환
강사: 대기 화면   → 학생: WaitingPage 전환
강사: 수업 종료   → 학생: 요약 카드 표시
```
- 강사 액션이 학생 화면에 즉시 반영되는가?
- 전환 사이에 빈 화면이나 깜빡임이 없는가?

### 스태프↔학생 상호작용
```
학생: 도움 요청 → 스태프: DM 알림 → 스태프: 응답 → 학생: 답장 표시
학생: 손들기     → 스태프: 손들기 목록 → 스태프: 도움 완료
학생: 수업 질문  → 스태프: 질문 목록 → 스태프: 1:1 답변 or 답변 완료
학생: 추천(좋아요) → 강사/스태프: 추천 수 실시간 업데이트
```
- DM 미읽 수가 학생 "도움" 탭에 정확히 표시되는가?
- 스태프가 응답하면 학생에게 즉시 알림이 가는가?
- 추천이 실시간으로 강사 화면에 반영되는가?

### 라이브 뷰 연동
```
강사: 질문 활성화/전환 → 라이브 뷰: 자동 추적
학생: 투표              → 라이브 뷰: 바 실시간 성장
강사: 게임 모드 전환     → 라이브 뷰: 게임 화면 표시
```
- 라이브 뷰가 강사 액션을 빠짐없이 따라가는가?
- 투표 시 바가 spring 물리로 부드럽게 움직이는가?

### 채팅 연계
```
공개 채팅: 학생+스태프+강사 전원 참여
운영 채팅: 스태프+강사만 (학생 안 보임)
1:1 DM: 학생↔스태프 (다중 DM 지원)
수업 질문: 전체 공개 + 추천 + 1:1 답변 분기
```
- 운영 채팅이 학생에게 절대 안 보이는가?
- DM 목록에서 여러 스태프와의 대화가 각각 분리되는가?
- 수업 질문에서 "1:1 답변" 클릭 시 DM이 자동 생성되는가?

### 끊기기 쉬운 엣지 케이스
- 학생이 투표 중 강사가 질문을 전환하면?
- 스태프가 DM 응답 중 학생이 새 DM을 보내면?
- 라이브 뷰에서 게임→질문 모드 전환이 매끄러운가?
- 세팅중 세션에 학생이 접속하면 대기 화면이 나오는가?
- 수업 종료 후 학생이 새로고침하면 요약 카드가 보이는가?

### 🔴 모드 전환 UX 연계 (즉시 수정 필요)
- **강사 모바일: 모드 선택 시 자동으로 결과 탭으로 이동** — 현재 모드 선택해도 "진행" 탭에 그대로 있어서 혼란. switchMode 호출 후 결과 탭으로 자동 전환해야 함
- **전자칠판(/live)에서 Q&A 보드 모드 지원** — 강사가 qaBoard 모드 선택 시 전자칠판에서도 Q&A 보드가 표시되어야 함. 현재는 퀴즈 결과만 보이는 상태
- **모드 전환 시 학생 화면 즉시 반영** — 강사가 모드 전환하면 학생 WaitingPage의 모드 인디케이터가 즉시 변경

### 🔴 게임 수리 & 개선 (v3 최우선)

> 게임은 수업 쉬는 시간에 학생들이 가장 기대하는 콘텐츠.
> 프로젝터에서 교실 전체가 보기 때문에 **크기, 디자인, 연출**이 매우 중요.

#### ✅ 완료된 게임 작업 (v3)
- 777 슬롯: DOM 기반 사이클링으로 전면 재작성 (freeze 해결)
- 룰렛: 가중치 scores 연결 + indigo 그라데이션 + 연속 스핀 각도 보정
- 핀볼(Plinko): 8행 핀 보드, 공정 확률, spring 물리, 반응형
- 경품 추첨: SlotReel DOM 조작 방식으로 freeze 수정
- 전체 게임: max-w-2xl 프로젝터 확대, 당첨자 text-5xl, confetti 32파티클
- 전체 게임: React 18 StrictMode mountedRef 버그 수정 (Plinko/SlotMachine/Roulette)
- 돌림판: 당첨 시 휠 축소(max-w-xs) → 당첨자+버튼 화면 안에 수용
- 쉬는 시간: 프리셋 버튼 dark pill 스타일 + BreakMascot transformOrigin 수정
- 팀 대항전: 별도 아코디언 → 모드 드롭다운 통합, 설정 UI 센터 표시

#### 게임 디자인 원칙
```
1. 크기 = 임팩트. 프로젝터 뒷자리에서도 보여야 함
2. 래퍼런스 리서치 필수 — Kahoot/Wheel of Names/Plinko 참고
3. 당첨 순간 = 클라이맥스 — 축하 애니메이션 + confetti + 이름 크게
4. 사운드 없이도 흥미 — 시각적 긴장감 + 스피드 변화로 연출
5. 색상은 디자인 시스템 내 — indigo 그라데이션 + slate
6. 강사가 "다시 돌리기" 쉽게 — 원클릭 리셋
```
